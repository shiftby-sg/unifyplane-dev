## Navigation Intent

The primary navigation should guide readers in a disciplined order:

**problem → definition → current truth → evidence → operational reality → deeper understanding**

This keeps first contact simple, grounded, and honest.

Readers should first understand:

* what problem UnifyPlane addresses
* what UnifyPlane is
* what is already real
* where real evidence exists
* what remains emerging or future-facing

Only after that should they move into deeper operational and conceptual material.

---

## Why This Structure

This navigation is designed to avoid three common problems:

* leading too early with abstract doctrine
* making the site feel like a framework catalog
* overstating maturity before evidence and readiness are clear

The public-facing site should feel:

* clear
* bounded
* evidence-led
* implementation-aware

That means the top navigation should begin with the problem and current truth, not with internal structure or advanced domain mapping.

It should also keep AI, safety, and security visible where relevant, without allowing them to replace the continuity-first identity of the site.

The fastest possible signal.

* the continuity problem

A fuller explanation of the continuity problem UnifyPlane responds to, grounded in real enterprise practice and current delivery reality.

### Current Readiness

The main trust page.

This page separates:

* what is proven now
* what is implemented but immature
* what is future but grounded

### Evidence

A guided view of real implementation and runtime evidence.

This page should explain what evidence exists, what it currently proves, and what it does not yet prove.

### Components

A clearer view of the current operational landscape, including:

* `unifyplane-core`
