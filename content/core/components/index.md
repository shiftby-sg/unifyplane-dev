## Components

This page explains the real operational parts of UnifyPlane.

Components are not audience lenses, abstract ideas, or enterprise disciplines. They are the current execution and inspection surfaces through which UnifyPlane already shows real behavior, real evidence, and real maturity boundaries.

At the current stage, the component landscape is centered on three parts:

* `unifyplane-core`
* `agent-runtime`
* `inspect-repo`

These components do not all have the same role, and they do not all have the same maturity. That difference matters.

---

## Why Components Matter

UnifyPlane is not best understood only as a concept or only as a high-level model.

Its current reality depends on actual operational parts that:

* carry declared inputs into runtime
* execute inside bounded structures
* emit structured evidence
* expose drift-oriented inspection surfaces
* make some forms of proof possible today

That means the Components layer is where the current implementation landscape becomes visible.

This page should be read together with:

* **What is UnifyPlane**
* **Current Readiness**
* **Evidence**

Those pages explain the broader model, maturity boundaries, and proof surfaces. This page explains where those realities currently live operationally.

---

## The Current Component Landscape

### `unifyplane-core`

The strongest and most mature current implementation surface.

`unifyplane-core` is the current execution and evidence center of UnifyPlane. It is strongest where systems need:

* declared-to-runtime continuity
* bounded execution
* evidence-backed runtime closure
* proof
* drift detection and drift reduction

It is also the place where different downstream component paths are currently coordinated under one bounded runtime model.

**How to read it:**  
Core is the strongest current implementation center, but it is not the whole of UnifyPlane.

[Read more: UnifyPlane Core](#)

---

### `agent-runtime`

A bounded AI-agent execution and assurance runtime.

`agent-runtime` shows how AI-agent participation can exist inside the broader UnifyPlane model without becoming open-ended or opaque. It already demonstrates:

* bounded AI-agent execution
* runtime-side evidence
* guardrailed execution behavior
* early AI-runtime governance and assurance relevance

Its current importance is not that it is already a broad mature AI-agent platform. Its importance is that it proves bounded AI-agent execution can already exist with evidence and runtime discipline.

**How to read it:**  
Agent Runtime is real and meaningful, but still early in breadth, maturity, and reusable namespace depth.

[Read more: Agent Runtime](#)

---

### `inspect-repo`

A bounded inspection component for repository, contract, and structural conformance surfaces.

`inspect-repo` is the clearest current inspection-oriented surface in UnifyPlane. It already demonstrates:

* bounded multi-tool inspection
* inspection-side evidence
* structural and contract drift visibility
* early repo-conformance discipline
* an emerging non-AI governance role

Its current strength is in making structural and contract-oriented expectations more inspectable and more evidence-bearing.

**How to read it:**  
Inspect Repo is real and useful now, but still uneven in semantic depth across individual inspection outputs.

[Read more: Inspect Repo](#)

## How These Components Differ

These components should not be read as interchangeable.

### Core
Core is the strongest current runtime and evidence center.

### Agent Runtime
Agent Runtime is the strongest current AI-agent participation surface.

### Inspect Repo
Inspect Repo is the strongest current inspection and repo-conformance surface.

They belong to the same broader UnifyPlane model, but they contribute in different ways.

---

## Relative Maturity

The current maturity profile is uneven by design and by evidence.

### Strongest current component
* `unifyplane-core`

### Real but earlier components
* `agent-runtime`
* `inspect-repo`

That does not make the earlier components unimportant. It means they should be described honestly.

The Components layer should therefore be read with the same discipline used elsewhere in the site:

* what is real should be stated plainly
* what is early should be stated carefully
* what is future-facing should not be presented as if it already exists

---

## What the Component Layer Already Shows

Taken together, the current components already show that UnifyPlane is:

* not only conceptual
* not only a static documentation model
* not only a future architecture direction
* not limited to one execution shape

The component landscape already demonstrates:

* bounded execution
* bounded inspection
* bounded AI-agent execution
* structured evidence
* different downstream component types under one broader runtime discipline
* early but real governance-relevant structure across both conventional and AI-enabled surfaces

That is one of the most important truths about the current state of UnifyPlane: the operational landscape is already real, but uneven in maturity.

---

## What This Layer Does Not Mean Yet

The existence of these components does **not** yet mean:

* every lifecycle surface is mature
* every downstream component is equally deep
* all inspection outputs are uniformly rich
* AI-agent execution is already broad and domain-complete
* UnifyPlane is already a finished governance platform

The Components page exists to make the current landscape legible, not to flatten all maturity boundaries into one broad claim.
